## Quick start

```bash
npm install
npx playwright install --with-deps

# Run everything
npm test

# Smoke tests only (fast, <2 min)
npm run test:smoke

# Desktop only
npm run test:desktop

# Mobile only
npm run test:mobile

# View HTML report
npm run report
```

## Setup: GitHub repository

1. **Push this folder** as a new GitHub repository.

2. **Enable GitHub Pages:**
   - Settings в†’ Pages в†’ Source: `gh-pages` branch, root `/`
   - Your dashboard will live at: `https://<your-org>.github.io/<repo-name>/`

3. **Add secrets** (Settings в†’ Secrets в†’ Actions):
   ```
   TEST_USER_EMAIL      real test account email
   TEST_USER_PASSWORD   real test account password
   ```

4. **First run:**
   - Push to `main` вЂ” the workflow triggers automatically.
   - Or: Actions в†’ "Bazaar Printing E2E Tests" в†’ Run workflow.
   - After ~5 min, the dashboard updates at your GitHub Pages URL.

## Schedule

The CI runs on three triggers:
- **Every 6 hours** (continuous monitoring)
- **Weekdays at 9 AM UTC** (2 AM LA time, pre-business-hours health check)
- **Every push to `main`** (regression guard)
- **Manual** from the Actions tab, with a suite selector dropdown

## Cross-device sync tests

`tests/e2e/cross-device-sync.spec.ts` runs on **all 6 Playwright projects** simultaneously. Each test in that file asserts that the same behavior works identically on Chrome, Firefox, Safari, iOS, Android, and iPad. Any failure tells you exactly which device broke.

Current sync test groups:
- `[Sync] Navigation` вЂ” logo, cart, category links reachable
- `[Sync] Homepage content` вЂ” heading text, CTAs
- `[Sync] Category pages` вЂ” product listings, navigation
- `[Sync] Product configurator` вЂ” loads without 404, no horizontal overflow
- `[Sync] Auth page` вЂ” form renders, invalid login shows error
- `[Sync] SEO & meta tags` вЂ” description, canonical URL

## Product configurator notes

The configurator is fully client-side rendered (Next.js). The selectors in `ProductPage.ts` include `[data-testid="..."]` targets that don't yet exist in the live DOM. After your first manual exploration session:

1. Inspect the rendered HTML of `/bazaarprinting/product/roll-labels?id=1`
2. Update the selectors in `tests/pages/ProductPage.ts`
3. Consider asking the dev team to add `data-testid` attributes to the configurator steps вЂ” this makes tests much more stable than class-based selectors.

## Dashboard

The dashboard at `dashboard/index.html` shows:
- **KPI cards** вЂ” total tests, passed, failed, pass rate, run count
- **Matrix table** вЂ” test name (rows) Г— run time (columns), green вњ“ / red вњ— dots
- **Device filter chips** вЂ” filter to see only one browser/device
- **Trend chart** вЂ” pass/fail count over the last 50 runs
- **Tooltips** вЂ” hover any dot for details
- **Auto-refresh** вЂ” reloads every 10 minutes

The dashboard reads `data.js`, which is regenerated by `scripts/collect-results.js` and committed to `gh-pages` after every CI run.
